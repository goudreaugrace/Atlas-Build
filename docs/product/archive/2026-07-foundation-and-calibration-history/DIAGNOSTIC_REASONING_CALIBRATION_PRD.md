# Diagnostic Reasoning Calibration PRD

## Purpose

This PRD captures the next major upgrade to BBE Brand Doctor after the July 2026 prototype review transcript and the Q1 2026 United States Snacks automated report intake.

The product has already proven that a conversational, Jarvis-style brand intelligence experience can feel compelling. The next risk is trust. The system must become visibly better at deciding what the data means before it creates more polished answers, decks, QBRs, or treatment recommendations.

The next phase is therefore a Diagnostic Reasoning Calibration phase, not a feature-expansion phase.

## Product Thesis

> The prototype should move from a compelling assistant with good data access to a governed diagnostic system whose data, definitions, reasoning, evidence, and outputs all agree.

The user-facing experience can remain magical. The underlying system must become explicit, testable, and source-governed.

This phase should also preserve the reusable architecture direction documented in `docs/product/REUSABLE_GOVERNED_INTELLIGENCE_SHELL_ARCHITECTURE.md`: BBE should become the first domain pack inside a governed intelligence shell, not a one-off hardcoded app.

## Triggering Learnings

### Prototype Review Transcript

Kate and Lydia's feedback exposed a core diagnostic risk:

- The system over-weighted category index and brand scale.
- It could imply Lay's was "strong" when the more useful read is that a large/category-leading brand may still be vulnerable.
- Momentum should be the headline verdict.
- Ahead/Behind is the size-adjusted check.
- Category index is useful context, not health proof.
- M/D/S and Pricing Power need to be interpreted as a connected profile, not isolated metric cards.
- Demographic-specific questions will come up, and the system should refuse or clearly caveat when the cut is unavailable.

### Q1 2026 Snacks Automated Report

The deck independently reinforces the same doctrine:

- `vs. Category`, `Ahead`, and `Momentum` are distinct lenses.
- Ambition is to be `Ahead` and in `Momentum`, not merely above category average.
- The dummy chart-read example shows a brand over-indexing versus category while not being Ahead on key measures and flat on Momentum.
- Demand Power and Pricing Power are outcome KPIs with different M/D/S driver weights.
- Pricing Power is broad equity-based price justification, not SKU-level pricing guidance.
- Report outputs are modular and could eventually be generated by the system if the data and reasoning pipeline are hardened.

## Goals

1. Harden the data foundation before expanding outputs.
2. Convert transcript and deck learnings into explicit definitions, rules, gates, and evals.
3. Add an Equity Reasoning Layer that produces the canonical diagnostic read used by chat, Jarvis, work artifacts, report views, and future deck modules.
4. Add simulated demographic diagnostics for prototype storytelling, while making replacement requirements unmistakable.
5. Build report/deck modules only after the underlying data and reasoning contracts are stable.
6. Make every important claim traceable to source evidence, missing evidence, or prototype simulation status.
7. Keep data, knowledge, reasoning, evidence, skills, and output modules separated so the same shell can later host another domain pack.

## Non-Goals

- Do not replace the full 128-slide automated report in the first pass.
- Do not treat simulated demographic data as measured evidence.
- Do not promote transcript feedback, extracted deck claims, or reviewed-local source candidates into canonical facts without source-owner governance.
- Do not let the LLM decide whether a brand is strong, vulnerable, improving, or deteriorating.
- Do not broaden pilot usage until core diagnostic reads pass golden tests.

## Primary Users

### Insights Leads

Need to inspect how the system reasoned, challenge the read, identify missing evidence, and calibrate the rule set.

### Brand Managers

Need a clear, plain-English diagnosis with enough proof to trust it and enough caveat to avoid misuse.

### Executive Sponsors

Need a concise decision read that is compelling but not analytically sloppy.

### Product / Methodology Owners

Need a system that can absorb updated doctrine, metric definitions, and approved data without whac-a-mole prompt edits.

## Functional Requirements

### 1. Canonical Data Foundation

The system must normalize BBE, deck, and future source data into inspectable records with provenance.

Required record fields include:

- brand
- market
- category
- period
- metric
- value
- benchmark basis
- `vsCategory`
- `ahead`
- `momentum`
- base size
- peer set / life-stage basis where available
- source file
- source slide / chart / table
- source type
- confidence / significance
- approval status
- replacement requirement if simulated or assumption-based

### 2. Equity Reasoning Layer

The system must compute a canonical `EquityReasoningRead` for each active brand.

It should include:

- headline verdict
- momentum read
- category context
- ahead/behind read
- size-adjusted strength assessment
- strength-language permission
- integrated M/D/S and Pricing Power driver read
- large-but-vulnerable state where applicable
- evidence gaps
- blocked claims
- treatment implications
- recommended next evidence
- source confidence and caveats

### 3. Language Permission Rules

The system must block or qualify language such as "strong," "healthy," "winning," or "star" when the full profile does not support it.

Required rules:

- Do not call a brand strong because it over-indexes versus category.
- Do not call a brand strong if key size-adjusted `Ahead` signals are weak.
- Do not hide red or deteriorating Momentum behind scale.
- Use Kantar typology language only as source context unless the user explicitly asks.
- Use Pricing Power language as broad equity-based price justification, not SKU-level pricing advice.

### 4. Demographic Evidence Gate

The system must distinguish between:

- official BBE demographic performance cuts
- simulated prototype demographic cuts
- external demographic context such as Census
- no demographic evidence

If a user asks about a demographic and only total-market data exists, the system must say it cannot diagnose that cut.

If simulated data exists, the system may answer only with a visible prototype caveat.

### 5. Simulated Demographic Diagnostics Pack

The system may include a prototype-only simulated demographic pack for demonstration.

The pack must:

- be clearly separated from measured BBE data
- include segment-level provenance
- carry replacement requirements
- never be mixed into official reads without disclosure
- be excluded from pilot or executive factual claims unless explicitly presented as simulated

### 6. Golden Reads And Evals

The system must include calibration tests based on:

- the Lay's demo transcript tension
- the deck's three benchmark lenses
- category-leading but not-ahead examples
- demographic question refusal/caveat behavior
- Pricing Power guardrails
- M/D/S driver interpretation

Representative eval prompts:

- "Is Lay's strong or just big?"
- "Why is category index not enough?"
- "How is Lay's doing with younger consumers?"
- "What evidence would you need to diagnose Gen Z performance?"
- "A brand over-indexes versus category but is not Ahead on Pricing Power or Different. What should the read be?"

### 7. Governed Output Modules

Once the reasoning layer is stable, outputs should be composed from governed modules.

Initial modules:

- benchmark lens explainer
- chart-read module
- executive verdict
- MDS dashboard read
- Demand Power deep dive
- Pricing Power deep dive
- Meaningful / Different / Salient deep dives
- demographic diagnostic read
- source readiness panel
- provocation questions
- evidence gaps and next proof path

## Success Criteria

The upgrade is successful when:

- The system no longer calls a brand strong based on category index alone.
- Lay's-style large-but-vulnerable reads are consistent across chat, Jarvis, report, and work artifacts.
- Demographic questions are answered, caveated, or blocked correctly based on available evidence.
- Simulated data is visibly labeled everywhere it appears.
- The reasoning layer can be inspected by an insights lead.
- Golden evals pass before broader pilot testing.
- The first governed 6-10 slide brand read can be generated from the same data and reasoning objects used by the app.

## Business Inputs Required

Before pilot, methodology and business owners should provide:

- approved definitions for each BBE metric
- official interpretation of `vs. Category`, `Ahead`, and `Momentum`
- peer-set and life-stage rules behind `Ahead`
- official trend/significance interpretation for Momentum
- demographic cut availability and readable-base thresholds
- approved words to use or avoid
- golden human reads for priority brands
- GCC or source-team comparison reports
- rules for when simulated prototype data can be shown
- source hierarchy when signals conflict

## Product Posture

Recommended language for stakeholders:

> The prototype experience is working. The next phase calibrates the diagnostic brain. We are turning expert feedback and source-report doctrine into governed data contracts, reasoning rules, evidence gates, and output modules so the system can move from impressive demo to trusted pilot.
