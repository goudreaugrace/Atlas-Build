# User Journey

## 1. Solution Home

The user enters through a clear product home.

Primary choices:
- Pick a brand.
- Open portfolio intelligence.
- Learn the BBE system.
- Inspect data/source readiness.

Outcome:
- The user understands that Brand Doctor combines fixed diagnostics, Jarvis-style interaction, and durable output workspaces.

## 2. Brand Home

The user lands on `/brand/[brandId]`.

Brand Home shows:
- Brand, category, market, period.
- Current diagnosis and short strategic read.
- Core metric bloodwork.
- Brand DNA / Strategic Context status.
- Entry points to Report, Jarvis, Work, Data, Assistant, and future specialist paths.
- Recent requested work and approved starter workspaces.

Outcome:
- The user knows what is currently true for the brand and what the solution can do next.

## 3. Traditional Report

The user opens `/brand/[brandId]/report` for the fixed diagnostic read.

The report still owns:
- Executive decision brief.
- BBE bloodwork.
- Diagnosis and evidence.
- Explanation lenses.
- Portfolio pattern context.
- Dialog With Data.
- Treatment Path To Test.

Outcome:
- The user can trust the core diagnosis without relying on a conversational interface.

## 4. Jarvis Interaction

The user opens `/brand/[brandId]/jarvis` for the premium interaction layer.

Jarvis supports:
- Voice or typed questions.
- Fast direct answers first.
- Approval before heavier governed work.
- Live work-building presence.
- Governed outputs in a dynamic work canvas.
- Follow-up questions against the active brand and current work.

Outcome:
- The user feels like an expert brand strategist is in the room, grounded by the Brand Doctor foundation.

## 5. Work Request

The user asks for heavier work, such as:
- Build a QBR read with proof.
- Create a treatment path to test.
- Build a learning path.
- Prepare an agency brief.
- Pressure-test the evidence.
- Inspect source readiness.

The system:
- Classifies the request.
- Answers what it can answer immediately.
- Offers or requests approval for governed work.
- Selects approved skills, ExperiencePlan templates, and dynamic views.
- Shows proof, gaps, guardrails, and review gates.

Outcome:
- The user sees the work being built and understands what is approved, missing, or gated.

## 6. Durable Work Item

The user opens `/brand/[brandId]/work` or `/brand/[brandId]/work/[workId]`.

Work items show:
- Source prompt.
- Audience and objective.
- Approved skill.
- Approved template.
- Approved views.
- Evidence/gap/review summary.
- Share/export posture.
- Links back to Jarvis, Report, Data, and Brand Home.

Outcome:
- The output has a home beyond the chat transcript.

## 7. Evidence And Data Inspection

The user opens `/brand/[brandId]/data`.

Data owns:
- Brand Health Record.
- Brand Intelligence Packet.
- Brand Strategic Context.
- Momentum source readiness.
- Source claims and reviewed-local source candidates.
- Rules, treatments, knowledge graph, and raw JSON.

Outcome:
- Insights leads and source owners can inspect exactly what the agent and report are allowed to use.

## 8. Continuous Improvement

The system logs:
- Unanswered questions.
- Outside-current-evidence asks.
- Requested work types.
- Proof gaps.
- Review gates.
- Local prototype transcripts for diagnostics.

Outcome:
- The skills library, data contracts, and approved view registry can expand from real user demand without weakening the guardrails.
